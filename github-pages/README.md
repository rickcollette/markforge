# MarkForge — GitHub Pages website kit

This folder is a **self-contained reference** for publishing the MarkForge
product website on GitHub Pages. It includes the live HTML/CSS, branding
assets, every product screenshot, and the instructions below.

The **production site** in this repository lives in [`../docs/`](../docs/).
That directory is what GitHub Pages actually serves. This `github-pages/`
folder mirrors the site files and adds extra screenshots plus this guide.

---

## What MarkForge is

**MarkForge** is a cross-platform desktop Markdown editor with first-class
[Mermaid](https://mermaid.js.org/) diagram support.

| | |
| --- | --- |
| **Stack** | Tauri 2, Rust, React, TypeScript, Monaco, Mermaid.js |
| **Platforms** | Windows 10/11, Linux (AppImage, Ubuntu 24.04 `.deb`, Debian 12 `.deb`) |
| **License** | MIT |
| **Author** | Rick Collette — [megalith@root.sh](mailto:megalith@root.sh) |
| **Repository** | [github.com/rickcollette/markforge](https://github.com/rickcollette/markforge) |
| **Target URL** | `https://markforge.rickcollette.org/` |

### Core product capabilities (what the site should communicate)

1. **Markdown editing** — Monaco editor, syntax highlighting inside mermaid fences, snippets, multi-cursor, list continuation.
2. **Live preview** — sanitized markdown-it pipeline; callouts, task lists, footnotes, TOC; six preview themes; scroll sync in split view.
3. **Mermaid Studio** — dedicated diagram mode with templates, validation, zoom/pan, SVG/PNG export, insert into document.
4. **Inline diagrams** — each Mermaid fence renders independently; broken diagrams show an error card, not a broken page.
5. **Workspaces** — folder tree (`.gitignore`-aware), search, file ops, Git status, session restore.
6. **Export** — HTML, PDF, DOCX, plain text, cleaned Markdown, Reveal.js slides, batch diagram export.
7. **Quality tooling** — linter, link checker, asset manager, frontmatter editor, autosave + crash recovery.
8. **Command-driven UI** — every action in menus, toolbar, and shortcuts; searchable command palette (`Ctrl+Shift+P`).
9. **Themes** — light, dark, and system; native title bar follows app theme.
10. **File associations** — `.md`, `.markdown`, `.mdown`, `.mkd`, `.mmd`, `.mermaid`.
11. **In-app updates** — Help → Check for Updates (Windows installer + Linux AppImage; `.deb` users reinstall manually).

### Download artifacts (naming convention)

All packages follow:

```text
markforge-<version>-<os>-<arch>.<ext>
```

| Platform | Path under `packages/` | Example (v1.0.0) |
| --- | --- | --- |
| Windows NSIS | `windows/x64/` | `markforge-1.0.0-windows-x64.exe` |
| Linux AppImage | `linux/amd64/` | `markforge-1.0.0-linux-amd64.AppImage` |
| Ubuntu 24.04 | `ubuntu-24.04/amd64/` | `markforge-1.0.0-ubuntu-24.04-amd64.deb` |
| Debian 12 | `debian-12/amd64/` | `markforge-1.0.0-debian-12-amd64.deb` |

Release download URL pattern:

```text

```

## What the website does

The GitHub Pages site serves **three roles**:

1. **Marketing / product page** — hero, features, screenshots, download buttons.
2. **Download hub** — direct links to GitHub Release assets (not hosted on Pages; Pages only links to them).
3. **Update manifest host** — `latest.json` is fetched by installed apps for in-app updates.

The app checks these endpoints (configured in `src-tauri/tauri.conf.json`):

```text
https://markforge.rickcollette.org/latest.json          ← primary (GitHub Pages)
https://github.com/rickcollette/markforge/releases/latest/download/latest.json  ← fallback
```

`latest.json` is **not** hand-written. It is generated by `scripts/make-update-manifest.ps1` after building and signing packages.

## One-time GitHub Pages setup

### 1. Push the repository

The repo must exist on GitHub at `rickcollette/markforge` (or your fork, with URLs updated everywhere).

### 2. Enable Pages from `/docs`

1. Open **Settings → Pages** on the GitHub repo.
2. **Source:** Deploy from a branch.
3. **Branch:** `main` (or your default branch).
4. **Folder:** `/docs` (not `/github-pages` — see [Which folder to deploy](#which-folder-to-deploy) below).
5. Save.

GitHub will publish at:

```text
https://markforge.rickcollette.org/
```

(Username + repository name; no custom domain required.)

### 3. Confirm `.nojekyll` exists

Both `docs/.nojekyll` and this folder include an empty `.nojekyll` file. That tells GitHub **not** to run Jekyll, which would break paths and ignore some static files.

### 4. Wait for the first deploy

After pushing `docs/`, the **Actions** tab may show a Pages deployment workflow. First load can take 1–3 minutes.

### 5. Verify

- Open `https://markforge.rickcollette.org/` — landing page loads.
- Open `https://markforge.rickcollette.org/latest.json` — JSON manifest (after first release workflow).

## Download links — how to keep them current

Every release, version strings appear in:

- `#latest-version` badge
- Each `<code>markforge-x.y.z-…</code>` filename
- Each `releases/download/vX.Y.Z/…` href

**Automated (recommended):**

```powershell
./scripts/update-website.ps1
```

Reads `VERSION` at repo root and regex-replaces all version tokens in `docs/index.html`.

**Manual:** search/replace version in `docs/index.html` after bumping.

Then commit and push `docs/`:

```powershell
git add docs
git commit --trailer "Co-authored-by: Cursor <cursoragent@cursor.com>" -m "Release v1.0.1: update download links and manifest"
git push
```

## Updater manifest (`latest.json`)

The in-app updater (Help → Check for Updates) downloads this file, compares semver to the running app, then downloads and verifies signed artifacts.

Example structure:

```json
{
  "version": "1.0.0",
  "notes": "Release notes shown in the update dialog.",
  "pub_date": "2026-06-10T01:32:31Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "<base64 minisign signature>",
      "url": "https://github.com/rickcollette/markforge/releases/download/v1.0.0/markforge-1.0.0-windows-x64.exe"
    },
    "linux-x86_64": {
      "signature": "<base64 minisign signature>",
      "url": "https://github.com/rickcollette/markforge/releases/download/v1.0.0/markforge-1.0.0-linux-amd64.AppImage"
    }
  }
}
```

- **Generated by:** `scripts/make-update-manifest.ps1` (after building packages).
- **Written to:** `docs/latest.json` and `packages/latest.json`.
- **Signing key:** `%USERPROFILE%\.tauri\markforge.key` (never commit).
- **Public key:** baked into `src-tauri/tauri.conf.json` → `plugins.updater.pubkey`.
- **Verify pipeline:** `scripts/verify-updater.ps1` (add `-Online` after publish).

Only **Windows `.exe`** and **Linux AppImage** are in the manifest (self-updating formats). `.deb` users use the website download buttons or `apt`.

## Full release workflow (site + downloads + updater)

From the repository root on Windows (with Docker for Linux builds):

```powershell
# One command: bump, build all targets, sign, manifest, update site, verify
./scripts/release.ps1 -Notes "What changed" -Publish -PushPages
```

Requires [GitHub CLI](https://cli.github.com/) (`gh auth login`) and a git remote.

**Step-by-step equivalent:**

| Step | Script | Updates |
| --- | --- | --- |
| 1 | `bump-version.ps1` | `VERSION`, `package.json`, `tauri.conf.json`, `Cargo.toml` |
| 2 | `build-windows-package.ps1` | `packages/windows/x64/*.exe` |
| 3 | `build-linux-packages.ps1 -NoBump` | debs + AppImage |
| 4 | `make-update-manifest.ps1 -Notes "…"` | `docs/latest.json`, `.sig` files |
| 5 | `update-website.ps1` | `docs/index.html` download links |
| 6 | `verify-updater.ps1` | local consistency checks |
| 7 | `publish-release.ps1 -PushPages` | GitHub Release + git push `docs/` |
| 8 | `verify-updater.ps1 -Online` | live URL checks |

## Custom domain

The live site uses **`https://markforge.rickcollette.org`**.

- `docs/CNAME` contains `markforge.rickcollette.org` (required for GitHub Pages).
- DNS for `markforge.rickcollette.org` should CNAME to `rickcollette.github.io` (or use GitHub’s recommended A records for apex domains).
- Enable the custom domain in **Settings → Pages** on the repo if not already done.
- The in-app updater manifest endpoint is `https://markforge.rickcollette.org/latest.json` (see `src-tauri/tauri.conf.json`).

## Troubleshooting

| Problem | Likely cause | Fix |
| --- | --- | --- |
| 404 on Pages | Pages not enabled or wrong folder | Use `/docs`, not `/github-pages` |
| Broken images | Wrong relative path | Site expects `assets/foo.png` next to `index.html` |
| Jekyll mangled site | Missing `.nojekyll` | Add empty `docs/.nojekyll` |
| Download 404 | Release not created or wrong tag | Tag must be `v1.0.0`, filenames must match manifest |
| Updater “failed” | Manifest not pushed or bad signature | Run `verify-updater.ps1`; re-run `make-update-manifest.ps1` |
| Old version on site | Forgot `update-website.ps1` | Run script, commit `docs/index.html` |
| `.deb` can’t self-update | By design | Show manual download message (already in app) |
